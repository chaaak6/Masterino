# 知识库 / 文件上传

MasterLion 支持文件上传 / 知识库管理。该功能依赖于以下核心技术组件，了解这些组件将有助于你成功部署和维护知识库系统。

## 核心组件

### 1. PostgreSQL 与 PGVector

PostgreSQL 是一个强大的开源关系型数据库系统，而 PGVector 是其扩展，为向量操作提供支持。

- **用途**：存储结构化数据和向量索引
- **部署建议**：使用 ParadeDB Docker 镜像可以快速部署包含 pgvector 和 pg\_search 插件的 PostgreSQL

示例部署脚本：

```
docker run -p 5432:5432 -d --name pg -e POSTGRES_PASSWORD=mysecretpassword paradedb/paradedb:latest-pg17
```

- **注意事项**：确保分配足够的资源以处理向量操作

### 2. S3 兼容的对象存储

S3（或兼容 S3 协议的存储服务）用于存储上传的文件。

- **用途**：存储原始文件
- **选项**：可以使用 AWS S3、RustFS、ceph 或其他兼容 S3 协议的存储服务
- **注意事项**：配置适当的访问权限和安全策略

如需做对象存储 A/B 诊断，可以在部署 compose 命令中额外加入
`docker-compose.rustfs.yml`，让新上传文件指向 `S3_ENABLE_PATH_STYLE=1` 的 RustFS。该方式不会
迁移 COS/OSS 里的历史对象；正式迁移需要把已有对象 key 复制到 RustFS，或实现双后端读取兜底。

### 3. 嵌入模型（Embedding）

嵌入模型用于将已解析的文本转化为向量表示，供语义检索使用。

<Callout type={'info'}>
  MasterLion 的历史兜底配置是 OpenAI `text-embedding-3-small`。在接入 Aihub 的部署中，
  请把 `DEFAULT_FILES_CONFIG` 设置为 Aihub 已授权给托管 token 或用户组的 embedding 模型。
  否则可能出现文件分块成功，但向量化任务以 `ModelNotFound` 失败。
</Callout>

- **用途**：生成文本的向量表示，用于语义搜索
- **注意事项**：
  - 需要配置的模型提供商里存在可用的 embedding 模型
  - Aihub 集成请使用 `newapi` provider 前缀，并填写 Aihub 实际授权的模型 ID
  - RustFS、MinIO、OSS、COS 等对象存储只负责保存文件，不提供解析、分块或向量化

### 4. Unstructured.io（可选）

Unstructured.io 是一个强大的文档处理工具。

- **用途**：处理复杂的文档格式，提取结构化信息
- **应用场景**：处理 PDF、Word 等非纯文本格式的文档
- **注意事项**：评估处理需求，根据文档复杂度决定是否部署

通过正确配置和集成这些核心组件，您可以为 MasterLion 构建一个强大、高效的知识库系统。每个组件都在整体架构中扮演着关键角色，共同支持高级的文档管理和智能检索功能。

### 5. 自定义 Embedding（可选）

- **用途**: 使用不同的嵌入模型（Embedding）生成文本的向量表示，用于语义搜索
- **选项**: 支持的模型提供商:newapi/zhipu/github/openai/bedrock/ollama
- **部署建议**: 使用环境变量配置默认嵌入模型

```
environment:
  DEFAULT_FILES_CONFIG: embedding_model=newapi/YOUR_AIHUB_EMBEDDING_MODEL,query_mode=full_text
  CHUNKS_AUTO_EMBEDDING: '1'
```

如果只需要上传与全文解析，可以保持 `CHUNKS_AUTO_EMBEDDING=0`；已经因模型不存在而失败的历史
embedding 任务，需要在修正模型配置后重新上传或重试。
